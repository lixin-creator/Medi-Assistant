# 附近医院查询按钮 + Hospital MCP 设计文档

**日期：** 2026-03-09
**目标：** 在现有项目中新增一个独立于聊天工作流的“查询附近医院”按钮，基于用户当前位置返回附近医院列表，展示医院等级、距离、地址、电话和导航链接。

## 一、问题定义

当前项目的核心能力是基于 LangGraph 的医疗问答。用户提出的新需求不是通过自然语言在聊天中触发“附近医院查询”，而是在前端提供一个明确按钮，点击后直接发起基于当前位置的医院查询。

这意味着该功能不应耦合进现有 LangGraph 对话链路，而应作为一个独立的产品能力实现。

## 二、设计目标

- 保持聊天工作流与位置查询功能解耦
- 用 MCP 作为医院查询能力的扩展层
- 第一版只解决“当前位置 -> 附近医院列表”的明确需求
- 输出结构化医院数据，便于前端稳定展示
- 对医院等级做统一标准化处理，避免不同数据源字段不一致

## 三、非目标

第一版不包含以下能力：

- 不接入 LangGraph 的 PlannerAgent / ExecutorAgent
- 不通过聊天输入触发医院查询
- 不做在线挂号、医生排班、科室推荐
- 不做急诊判断或智能分诊
- 不做用户历史位置保存
- 不做后台静默定位

## 四、总体架构

```text
前端按钮“查询附近医院”
        │
        ▼
浏览器 Geolocation API
        │
        ▼
POST /api/v1/hospitals/nearby
        │
        ▼
HospitalService
        │
        ▼
HospitalMcpClient
        │
        ▼
Hospital MCP Server
        │
        ▼
地图 / POI / 医疗机构数据源
        │
        ▼
结构化医院列表
        │
        ▼
前端卡片列表渲染
```

## 五、职责划分

### 1. 前端

负责：

- 提供“查询附近医院”按钮
- 请求浏览器定位权限
- 调用后端查询接口
- 展示加载状态、错误状态、结果列表
- 渲染医院名称、等级、距离、地址、电话、导航链接

不负责：

- 直接调用地图平台 API
- 直接调用 MCP Server
- 做医院等级归一化

### 2. 后端 API

负责：

- 提供 `/api/v1/hospitals/nearby` 接口
- 校验经纬度、半径、返回数量等参数
- 调用 `HospitalService`
- 统一返回前端友好的结构

### 3. HospitalService

负责：

- 调用 `HospitalMcpClient`
- 对医院等级字段做标准化
- 对距离字段做统一格式化
- 屏蔽底层 provider/MCP 差异

### 4. HospitalMcpClient

负责：

- 封装 MCP 请求逻辑
- 向上暴露稳定接口，例如 `search_nearby_hospitals()`
- 不包含业务展示逻辑

### 5. Hospital MCP Server

负责：

- 暴露医院查询能力
- 对接外部地图/POI 数据源
- 返回原始或半标准化地点数据

## 六、为什么不走 LangGraph

该功能是一个显式 UI 操作，不是对话推理任务。将其硬塞进 LangGraph 会导致：

- 聊天职责膨胀
- Planner 路由逻辑复杂化
- 前端交互路径变长
- MCP 的价值无法体现为独立能力层

因此按单一职责原则，应将“问答工作流”和“位置服务工作流”分离。

## 七、接口设计

### 请求接口

`POST /api/v1/hospitals/nearby`

请求体：

```json
{
  "latitude": 31.2304,
  "longitude": 121.4737,
  "radius_meters": 5000,
  "limit": 10
}
```

### 响应体

```json
{
  "success": true,
  "hospitals": [
    {
      "id": "hospital_001",
      "name": "复旦大学附属中山医院",
      "level": "三级甲等",
      "distance_meters": 1200,
      "distance_text": "1.2 km",
      "address": "上海市徐汇区枫林路180号",
      "phone": "021-xxxxxxx",
      "latitude": 31.2,
      "longitude": 121.4,
      "navigation_url": "https://..."
    }
  ]
}
```

## 八、MCP 工具定义

第一版只保留一个核心工具，避免过度设计：

### `search_nearby_hospitals`

输入：

- `latitude`
- `longitude`
- `radius_meters`
- `limit`

输出：

- 医院列表
- 每个医院至少包含名称、地址、距离、原始等级字段、电话、坐标、导航链接

第二版可再增加：

- `get_hospital_details`

但第一版不是必需项。

## 九、医院等级标准化

不同外部数据源的“医院等级”字段可能不一致，因此后端必须增加标准化层。

建议统一输出如下枚举：

- `三级甲等`
- `三级乙等`
- `三级`
- `二级甲等`
- `二级`
- `一级`
- `未标注`

标准化策略：

- 优先读取数据源明确等级字段
- 若存在别名、简称或自由文本，则做映射归一
- 无法识别时统一返回 `未标注`

## 十、前端交互设计

### 入口

在主界面增加一个独立按钮：

- 中文：`查询附近医院`
- 英文：`Find Nearby Hospitals`

### 点击流程

1. 用户点击按钮
2. 浏览器请求定位权限
3. 成功获取经纬度后调用接口
4. 前端显示 loading
5. 收到结果后展示医院列表
6. 若失败则显示明确错误文案

### 列表字段

每条卡片展示：

- 医院名称
- 医院等级
- 距离
- 地址
- 电话
- 导航按钮

### 排序

- 默认按距离从近到远
- 第一版不增加复杂筛选

## 十一、错误处理

### 前端错误

- 用户拒绝定位权限：提示开启定位权限
- 浏览器不支持定位：提示手动升级浏览器或后续扩展手动输入位置
- 查询超时：提示稍后重试

### 后端错误

- 参数缺失或超范围：返回 422
- MCP 查询失败：返回 503 或统一业务错误
- 数据源缺失部分字段：返回空值或 `未提供`，不要整体失败

## 十二、安全与合规边界

- 必须由用户主动点击按钮后再请求定位
- 不缓存或持久化用户位置，除非后续有明确需求
- 明确这是“附近医院列表”，不是医疗推荐结论
- 不输出“最佳医院”“最权威医院”之类带决策引导的话术

## 十三、测试策略

### 后端

- 接口参数校验测试
- `HospitalService` 等级标准化测试
- `HospitalService` 距离格式化测试
- MCP client 异常降级测试
- endpoint 正常返回结构测试

### 前端

- 点击按钮后请求定位
- 定位成功后触发查询请求
- 结果列表渲染测试
- 定位失败错误提示测试
- loading 状态测试

## 十四、最小可交付版本

第一版完成标准：

- 有按钮
- 能获取当前位置
- 能返回附近医院列表
- 能展示等级、距离、地址、电话、导航链接
- 不影响现有聊天功能
